Security & deployment

Evaluate Blackridge inside an explicit technical boundary.

Blackridge can begin from existing evidence or observe a scoped workflow at runtime. Deployment, access, storage, retention, and failure behavior are agreed with each design partner rather than hidden behind a generic “enterprise ready” claim.

Body-free economics records by default · Customer-controlled perimeter · Explicit retention and failure tradeoffs

Evaluation modes

Begin without changing traffic, or observe one workflow inline.

The lowest-risk path depends on the evidence already available and the confidence required from the assessment.

Import assessment

No initial request-path change

Northwatch can analyze supported logs, traces, gateway exports, provider usage records, and operational metadata through an agreed secure handoff. Raw source exports may contain sensitive information and must be protected before and during normalization.

Runtime observation

Scoped production-adjacent deployment

For supported request paths, a selected workflow can run through Blackridge in an observe-oriented configuration. The workflow boundary, upstream providers, evidence storage, access controls, and success criteria are agreed before traffic moves.

Data boundary

Economic evidence is designed to avoid prompt and response retention.

An inline gateway necessarily handles a request payload in order to forward it. Blackridge does not persist prompt or response body content in default economics records or expose that content as operator evidence.

Default economics evidence
Request and correlation identifiers.
Provider, model, timing, status, and usage context.
Pricing and cost context where available.
Route, cache, attempt, fallback, and workflow relationships where available.
Tenant, application, team, user, and workflow attribution when supplied.
Do not place in evidence metadata
Prompts, instructions, messages, tool content, or response bodies.
Provider API keys, bearer tokens, or application secrets.
Customer documents or raw personally identifiable information.
Sensitive values that can be represented by stable identifiers and joined inside customer-controlled systems.
Security responsibilities

Blackridge is deployed behind customer-controlled security boundaries.

The current runtime is an infrastructure component, not a public SaaS endpoint. Production hardening depends on the surrounding environment and must be reviewed as part of deployment scope.

Storage & retention

Evidence location and lifecycle are part of the deployment decision.

Blackridge currently supports file-backed and database-backed economics-event storage. Retention is not silently managed by a hosted Blackridge service; it must be defined operationally for each environment.

Event storage

Assessment deployments can use append-only file storage. Database-backed storage is the stronger fit for durable or multi-instance collection.

Retention and deletion

Retention for events, persistent volumes, database rows, imported raw records, and exported reports must be agreed and implemented as part of the deployment process.

Imported records

Raw exports may contain fields that are excluded from normalized economics evidence. They require separate protection, access, and deletion handling.

Failure behavior

Inference continuity and evidence durability are explicit tradeoffs.

Runtime evidence writes are asynchronous by default so event I/O does not block model responses. That choice protects the request path but means evidence can be dropped when queues or sinks fail.

Default continuity posture

Event writes default toward fail-open behavior. Inference continues when an asynchronous evidence queue cannot accept another event.

Visible evidence gaps

Queue depth, dropped events, sink errors, and flush behavior are measurable. Reports should disclose incomplete evidence rather than imply full coverage.

Stricter collection

A fail-closed queue-full mode can be configured, but asynchronous sink failures can still occur after the response path. The desired posture must be tested for the scoped workflow.

Deployment & operations

Current deployment surfaces are assessment-oriented and observable.

Blackridge can run locally, in containers, or through the current Kubernetes assessment chart. This is not a claim of a general-purpose operator or turnkey production platform.

Container and Kubernetes

The current Kubernetes surface includes an assessment-oriented deployment, service account, service, persistent storage, pricing configuration, resource settings, and health probes.

Health and observability

Blackridge exposes liveness and readiness signals, structured logs, Prometheus metrics, and optional OpenTelemetry export for deployment monitoring.

Scaling boundary

File storage on a single persistent volume is not a horizontally scalable shared event store. Durable multi-instance designs should use an appropriate database-backed deployment.

Design-partner process

The website form is only for initial workflow context. Do not submit logs, credentials, prompts, or production data through it. If an assessment is accepted, Northwatch and the design partner agree the scope, transfer path, deployment boundary, retention, access, and start date separately.
