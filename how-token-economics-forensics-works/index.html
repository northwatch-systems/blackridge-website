Technical methodology

How Token Economics Forensics Reconstructs AI Workflow Spend

Blackridge treats every spend finding as an evidence problem. A finding should point back to the request, event, workflow, model, token, pricing, attribution, and confidence context that produced it.

Why provider bills are insufficient

Provider dashboards are useful for invoices, aggregate tokens, and model-level usage. They usually do not explain which workflow caused a spike, which retry loop multiplied cost, which fallback changed unit economics, or which app, team, tenant, or customer should own the spend.

What data Blackridge can ingest

Blackridge can work from logs, traces, gateway exports, provider usage data, runtime metadata, and application-level ownership signals. When higher-fidelity request evidence is required, Blackridge can collect events inline through an optional runtime collector or OpenAI-compatible collection path.

inputs:
  - provider_usage_export
  - gateway_request_log
  - otel_trace
  - app_log
  - app_team_tenant_metadata
outputs:
  - spend_map
  - waste_findings
  - attribution_coverage
  - evidence_rows

How workflow reconstruction works

Blackridge groups events into workflow boundaries using explicit identifiers where available and conservative inference where they are missing. It then reconstructs request lineage, model choices, token usage, retries, fallbacks, branch execution, and pricing assumptions for the scoped workflow.

group events by workflow_id or inferred workflow boundary
identify repeated calls with shared parent request, similarity, or retry metadata
compare observed spend against modeled baseline
attach confidence based on attribution and event completeness
emit finding with evidence rows

How attribution coverage and confidence work

Attribution is only as strong as the available metadata. Blackridge reports coverage gaps instead of pretending attribution is perfect. A finding can be exact, strong, inferred, weak, or unknown depending on the completeness of request IDs, workflow IDs, team/app/tenant metadata, pricing data, and event lineage.

Retry amplification

Retry amplification is detected when one user or workflow action produces repeated model calls that materially increase cost. Blackridge looks for retry metadata, repeated parent requests, repeated prompts or workflow steps, repeated failures, and cost variance against the expected baseline.

Fallback tax

Fallback tax appears when failures or routing decisions move requests to a more expensive model or provider. The finding should show the original path, fallback path, model choices, pricing assumptions, and the cost delta that changed unit economics.

Duplicate inference

Duplicate inference is flagged when similar prompts, repeated requests, or semantic duplicates are paid for repeatedly. Blackridge identifies candidates and labels confidence based on available request fingerprints, workflow context, and cache verdicts.

Premium-model overuse

Premium-model overuse is identified when expensive models appear to be defaults for tasks that may be served by cheaper models. Blackridge does not assume replacement is safe; it produces evidence so engineering can review model choice, task class, quality requirements, and cost impact.

Abandoned branch waste

Agent workflows may call models or tools along branches that never influence the final answer. Blackridge looks for branch lineage, tool invocation, final-response paths, and costs attached to abandoned execution branches.

Why findings must include evidence

A spend finding should be inspectable. The report should include supporting rows, source events, pricing assumptions, attribution coverage, and confidence notes so engineering and finance can evaluate the claim together.

Limits

Incomplete logs, missing request IDs, weak attribution metadata, sampling, privacy filtering, and provider export gaps can limit confidence. Blackridge reports those gaps and distinguishes observed waste from modeled opportunity.

Example synthetic investigation

A support-agent workflow shows a spend spike. Blackridge reconstructs the workflow and finds repeated failed calls followed by fallback to a premium model. Attribution is strong for app and team, weaker for customer because fallback events lack the customer header. The report ranks retry fanout first because the finding has clear event evidence and an engineering fix path.
