How BRG evaluates AI workflow spend

This page explains the evidence standard behind BRG workflow evaluation without exposing proprietary detection logic. The goal is simple: engineering and finance should be able to inspect why a runtime economics finding was made, what data supports it, and where confidence is limited.

What the evaluation is designed to answer

Provider dashboards are useful for invoices, aggregate tokens, and model-level usage. A BRG workflow evaluation is designed for a different review: which workflow drove spend, which ownership signals are present, what looks wasted, which controls are safe to test, and whether BRG should be deployed for ongoing control.

What Blackridge can use

BRG can work from existing logs, traces, gateway exports, provider usage data, runtime metadata, and app/team/tenant ownership signals. When existing data is incomplete, a scoped runtime collection path may be used to improve evidence quality.

Common inputs

Provider usage exports, gateway request logs, traces, application logs, pricing context, and ownership metadata where available.

Evidence outputs

A workflow spend map, waste findings, attribution coverage notes, confidence labels, safe tests, BRG fit decision, and evidence rows engineering can inspect.

The evidence standard

Each finding should include the supporting context needed to review it: affected workflow, request or event references, model and pricing assumptions, ownership metadata, coverage notes, and confidence. Weak or missing data should be visible in the evidence instead of hidden behind a chart.

Observed vs modeled

BRG separates observed waste from modeled opportunity. Observed findings are tied to available events. Modeled opportunity is presented as an estimate that engineering should validate before treating it as savings.

Coverage and confidence

Attribution is only as strong as the available metadata. BRG reports coverage gaps so teams can tell when a finding is well-supported, when it is inferred, and when better instrumentation is needed.

Waste classes covered in review

BRG evaluation can cover retry amplification, fallback tax, duplicate inference, premium-model overuse, abandoned branch waste, failed request waste, cache-miss opportunity, context bloat, retrieval bloat, and unattributed spend when the source data supports those findings.

What the evaluation does not claim

BRG evaluation does not guarantee savings, replace engineering judgment, or claim perfect attribution when metadata is incomplete. It gives engineering and finance structured evidence for deciding what to test and whether ongoing BRG deployment is justified.

Example review surface

A typical finding includes the affected workflow, the spend pattern, confidence, attribution coverage, observed vs modeled impact, supporting rows, and the recommended engineering test. The public example uses synthetic data to show the artifact without exposing customer data.
