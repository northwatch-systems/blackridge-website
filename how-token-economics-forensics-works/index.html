How it works

How Blackridge turns AI runtime activity into economic evidence.

Blackridge connects model requests to the applications, teams, tenants, and workflows that created them, reconstructs the runtime behavior behind their cost, and gives engineering an evidence trail for deciding what to change.

Two evidence paths · Body-free economics records by default · Observed and modeled results kept separate

Two ways to begin

Start from existing evidence or observe one workflow at runtime.

Both paths produce the same kind of economic evidence. The right starting point depends on the data already available and the level of confidence the evaluation requires.

Import assessment

Use evidence you already have

Blackridge can normalize supported logs, traces, gateway exports, provider usage records, pricing context, and ownership metadata. This is the lowest-friction way to evaluate one workflow without first changing its request path.

Runtime observation

Observe closer to the request path

For supported request paths, a scoped workflow can run through Blackridge so request context, provider outcomes, usage, cost, retries, fallbacks, and lineage can be recorded closer to where they occur.

Runtime architecture

The request path stays separate from the economics work.

In runtime mode, Blackridge sits between a supported AI application and model provider. The provider response returns to the application while economic evidence is recorded for off-path reconstruction and review.

01Application or agentSends a supported model request with available workflow and ownership context.
→
02BlackridgeForwards the request and records body-free economic evidence by default.

03Model providerReturns model output, usage, timing, and provider outcome where available.
Off-path economicsRequest ledger → workflow reconstruction → confidence-qualified findings → engineering recommendations
By default, Blackridge economics records exclude prompt and response bodies. They retain the operational metadata needed for economic review, such as request identity, provider and model context, timing, usage, cost, attribution, and workflow relationships when available.
Evidence process

From request activity to an engineering decision.

The public process is intentionally high level. Blackridge exposes the evidence behind a conclusion without publishing its internal schemas, scoring formulas, or detection implementation.

Collect or import

Use runtime observations or normalize supported existing records into a consistent economic evidence stream.

Connect ownership

Associate requests with tenants, applications, teams, users, and workflows where reliable metadata exists.

Reconstruct execution

Connect requests, attempts, retries, fallbacks, branches, provider outcomes, usage, and timing into workflow context.

Establish economics

Apply available pricing and usage context while keeping estimates, provider-reported values, and unknowns distinct.

Qualify findings

Separate observed behavior from inferred or modeled opportunity and expose coverage limitations.

Prioritize action

Present the evidence, likely impact, confidence, and engineering changes worth validating first.

Evidence standard

Blackridge distinguishes what it knows from what it estimates.

Missing pricing, attribution, usage, or lineage does not become a zero or a confident claim. Coverage and unknowns remain visible so engineering can judge the strength of each conclusion.

Observed

Directly supported by available runtime or imported evidence.

Inferred

Derived from available relationships with confidence and limitations shown.

Modeled

An opportunity estimate that still requires engineering validation.

Unknown

Evidence is insufficient; the missing coverage is reported rather than guessed.

Where Blackridge fits

Runtime economics complements the infrastructure you already operate.

Blackridge overlaps with parts of gateways, observability, and cost tooling, but its primary responsibility is different: preserve enough request and workflow evidence to explain why AI spend happened and what engineering should validate next.

Current scope

Observe, explain, and recommend first.

Blackridge is designed for production infrastructure, but public claims remain narrower than the full roadmap. Current evaluations focus on establishing trustworthy economic evidence before broader controls.

Buyer-safe current scope
Import supported evidence or observe scoped OpenAI- and Anthropic-compatible request paths.
Record body-free economics metadata by default rather than prompt or response content.
Attribute cost and runtime behavior where ownership and workflow signals exist.
Reconstruct retries, fallbacks, branches, usage, pricing context, and data-quality gaps where supported by evidence.
Produce request-level evidence, confidence-qualified findings, and ranked recommendations.
Explicit boundaries
Blackridge does not translate arbitrary provider request formats or claim universal provider compatibility.
Optional cache, routing, or budget behavior is not assumed to be enabled in an assessment.
Modeled opportunity is not reported as realized savings.
Incomplete metadata reduces attribution and lineage confidence.
Blackridge does not replace engineering judgment or guarantee savings.
