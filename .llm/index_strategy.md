# index_strategy.md

## Purpose

The homepage exists to create business urgency around Runtime Economics.

It is not a product page.

It is not a technical architecture page.

It is not a founder biography page.

Its purpose is to help technical leaders recognize that AI inference is becoming operational infrastructure and that they may already have visibility and attribution gaps.

The page should create enough belief and curiosity that visitors choose to either:

1. Review the Runtime Architecture
2. Request an Assessment

---

## Visitor

Primary visitors:

* CTOs
* VP Engineering
* Platform Leaders
* Infrastructure Leaders
* Technical Founders
* FinOps Stakeholders

Secondary visitors:

* Staff Engineers
* Principal Engineers
* Architects

---

## Visitor State On Arrival

The visitor likely:

* Knows AI is important
* Has experimented with AI
* Has heard concerns about growing AI spend
* Uses provider dashboards
* Has not yet thought deeply about runtime economics

The visitor does NOT yet know:

* Why runtime economics matters
* Why provider dashboards are insufficient
* Why a runtime boundary is valuable
* Why Blackridge exists

---

## Visitor State On Exit

The visitor should leave believing:

1. AI inference is becoming infrastructure.
2. Runtime economics is a real operational concern.
3. Provider dashboards cannot answer all runtime questions.
4. Visibility and attribution gaps likely exist.
5. Blackridge understands this problem.
6. Reviewing the architecture or requesting an assessment is worthwhile.

---

## What This Page Must Explain

* The pattern
* The visibility gap
* The attribution problem
* The business impact
* The value of a runtime boundary

---

## What This Page Must NOT Explain

* Hot path vs off path
* Request lifecycle
* Failure philosophy
* Runtime topology
* Deployment details
* Kubernetes
* Cache implementation
* Governance architecture

Those belong on the architecture page.

---

## Trust Signals

The page should establish trust through:

* Clear thinking
* Operational insight
* Recognition of common pain points
* Practical language

Not through:

* Founder names
* Technical jargon
* Product feature lists
* Hype
